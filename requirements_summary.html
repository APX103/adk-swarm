多智能体（Agent）协作平台需求总结

基于 Google ADK / A2A 生态构建“主 Agent + 专业子 Agent”的分布式智能体系统
版本 1.0 · 2026-06-16

1. 愿景一句话

打造一个以主 Agent 为人机交互门面、以多个专业 Agent 为能力单元的协作平台： 主 Agent 能够理解用户意图，自主决定调用哪些子 Agent（本地或远程、并发或链式、单轮或多轮）， 并在一个可记录、可观测、可扩展的平台上完成 Agent 之间的发现、通信与结果汇总。

2. 核心角色定义

主 Agent（Orchestrator / Facade）

用户日常聊天的唯一入口。具备强大的通用能力、规划能力、总结能力，并了解所有可用子 Agent 的技能范围。

专业子 Agent（Specialist Agents）

每个 Agent 拥有独立技能（Skills）、工具（Tools）与上下文。可由团队内部实现，也可由外部专家/第三方实现并接入。

平台层（Agent Interaction Platform）

负责 Agent 注册与发现、Agent 间通信转发、消息/任务记录、审计、鉴权、观测与人机协同入口。

人（Human-in-the-loop）

在关键决策、审批、异常、需要补充信息时介入；平台需提供可插拔的人工确认机制。

3. 功能性需求

3.1 主 Agent 必须具备的能力

  • 多工具调用：可同时调用多个工具并组合结果；支持并行 Function Calling。
  • Skills 体系：按场景加载 Skill（SKILL.md），实现上下文按需加载与能力模块化。
  • 上下文自管理：基于 Session / State / Memory 维护多轮上下文，并支持上下文压缩（Compaction）。
  • 流式输出：支持文本、音频、视频等流式响应，提升交互体验。
  • 规划与编排：能根据任务复杂度选择“直接回答 / 调用单 Agent / 并发调用多 Agent / 链式调用 / 让 Agent 之间多轮协商”。
  • 结果汇总：对子 Agent 返回的结果进行去噪、综合、格式化后返回给用户。

3.2 子 Agent 必须具备的能力

  • 拥有独立的 Skills + Tools 组合，聚焦一个或几个专业领域。
  • 拥有独立的 上下文空间，不被主 Agent 的上下文污染。
  • 可运行在独立的进程、容器或 Pod 中，实现运行环境隔离。
  • 能够以标准协议对外暴露能力，供主 Agent 或其他 Agent 发现和调用。

3.3 平台层必须具备的能力

  • Agent 注册与发现：维护 Agent Card / Skill 目录，让主 Agent 知道“谁能做什么”。
  • 通信中介：转发 Agent 间请求与响应；支持同步、流式、异步三种模式。
  • 消息记录与审计：可选地持久化 Agent 间通信内容，用于追溯、调试、合规。
  • 人机协同：在任务需要时暂停并通知人，接收人工输入后继续执行。
  • 鉴权与安全：Agent 间通信需身份校验、最小权限、加密传输。
  • 观测与评估：链路追踪、日志、指标、Agent 效果评估。

4. 非功能性需求

维度要求
稳定性单 Agent 故障不应拖垮整个系统；具备超时、重试、降级机制。
扩展性新 Agent 接入只需注册 Agent Card 并满足通信协议，无需改动主 Agent 代码。
隔离性子 Agent 可独立部署、独立伸缩、独立更新；不同 Agent 的依赖/环境互不干扰。
多模型不同 Agent 可选用不同模型（Gemini / GPT / Claude / 开源模型等）。
低延迟本地调用优先走内存/进程内；远程调用通过 A2A / gRPC / HTTP 减少开销。
可解释性每次决策应可追溯：为什么选这个 Agent、调用了什么、中间结果是什么。

5. 关键使用场景

场景 A：直接回答
低复杂度

用户问通用问题，主 Agent 直接调用自身工具 / Skills 回答。

场景 B：单专业 Agent
中等复杂度

主 Agent 识别到专业领域，调用对应远程 Agent 获取结果并汇总。

场景 C：并发专家会诊
高复杂度

一个问题需要法律、财务、技术三个 Agent 同时输出意见，主 Agent 综合三方结论。

场景 D：链式工作流
高复杂度

前一个 Agent 的输出作为后一个 Agent 的输入，按顺序完成数据提取 → 分析 → 生成报告。

场景 E：Agent 间自主协商
高复杂度

多个 Agent 围绕同一任务多轮沟通，平台记录完整协商过程，最终由主 Agent 给出结论。

场景 F：人工审批
合规场景

涉及金额、敏感操作或不确定结论时，暂停流程等待人工确认。

6. 已确认的关键决策

说明：经与用户确认，以下决策将指导后续实现。
  • 通信记录：不记录 Agent 间完整 Message 正文；仅在主 Agent 侧记录“调用了谁、返回了什么结果”等 Task 级元数据，便于追溯与调试。
  • 运行时隔离：不同 Agent 运行在不同 Container 中，通过 HTTP/A2A 通信；是否部署到独立 Pod 保持灵活，初期以独立 Container 为主。
  • 外部接入协议：强制使用 A2A 协议，保证开放性与未来生态兼容性。
  • 编排策略：以模型自主决策为主，同时提供预定义 Workflow 模板作为补充（混合模式）。
