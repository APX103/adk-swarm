Google ADK / A2A 技术调研报告与实现计划

评估 ADK 对“主 Agent + 多专业子 Agent”分布式协作平台的支撑度，并给出落地路线图
版本 1.0 · 2026-06-16

0. 执行摘要

经过对 Google Agent Development Kit (ADK)、 Agent2Agent (A2A) 协议 及相关生态的调研，结论如下：

  • 高度适配：ADK 原生支持多工具调用、Skills、流式输出、Session/State/Memory、上下文压缩、多 Agent 编排与 A2A 通信，能够作为整个系统的核心 Agent 构建框架。
  • 需要自研补充：ADK 本身不提供“Agent 间通信记录平台”、“集中式 Agent 注册中心”、“强制运行时隔离”和“企业级跨 Agent 鉴权”。这些能力需要在 ADK 之上构建一层平台层（Platform Layer）。
  • 推荐路线：采用 ADK + A2A + 自研 Platform Gateway 的混合架构：ADK 负责实现每个 Agent 的内在能力；A2A 负责跨进程/跨组织的 Agent 通信标准；自研平台负责注册发现、消息转发、审计日志、鉴权、人机协同与部署编排。

2. 关键模块深度调研

2.1 Agent 与工具模型

ADK 中的 Agent（或 LlmAgent）由三大件组成：

  • Model：默认 Gemini，也可通过 LiteLlm 接入 GPT/Claude/开源模型。
  • Instruction：系统提示，决定 Agent 的行为与边界。
  • Tools：可被 LLM 调用的能力单元。
关键设计：当主 Agent 配置 sub_agents=[...] 时，ADK 会自动把子 Agent 的 description 暴露给主 Agent，主 Agent 可自主决策把用户请求 transfer_to_agent 给某个子 Agent。

2.2 Skills 体系

ADK Skills 是“可热插拔的能力包”，推荐目录结构如下：

skills/
  example-skill/
    SKILL.md          # 必需：frontmatter + instructions
    references/       # 补充文档
    assets/           # 模板、数据、图片
    scripts/          # 可执行脚本

通过 load_skill_from_dir() 加载后，再用 SkillToolset 注入 Agent。这种“按需加载”机制正好符合用户对“上下文压缩、技能可插拔”的诉求。

2.3 流式输出

ADK 提供两种流式模式：

  • SSE（Server-Sent Events）：基于标准 Gemini API，适合文本流式返回。
  • BIDI（Bidirectional Streaming）：基于 Gemini Live API / WebSocket，支持文本、音频、视频实时输入输出。

核心抽象是 LiveRequestQueue（上行）和 Runner.run_live()（下行），可轻松包装成 FastAPI WebSocket 服务。

2.4 Session、State、Memory 与上下文压缩

  • Session：一次对话线程；由 SessionService 管理，支持内存、数据库、Vertex AI 三种实现。
  • State：每个 Session 内的键值对“草稿本”，通过 output_key、工具、回调更新。
  • Memory：跨 Session 的长期记忆，可用 InMemoryMemoryService 或 VertexAiMemoryBankService。
  • Context Compaction：通过 EventsCompactionConfig 自动对历史事件做 LLM Summary，非阻塞执行。
注意：上下文压缩目前为实验性功能，且对 SequentialAgent 等 Workflow Agent 需要显式指定 summarizer。生产环境建议先以自定义压缩策略兜底。

2.5 多 Agent 编排与 Workflow

ADK 把“多 Agent”视为 Workflow，主要编排原语：

原语用途
Agent / LlmAgent具备 LLM 推理能力的节点
SequentialAgent顺序执行多个子 Agent
ParallelAgent并发执行多个子 Agent，常用于 Fan-out
LoopAgent循环执行直到满足终止条件
CustomAgent（继承 BaseAgent）用确定性代码控制流程

Workflow 之间的通信主要依靠共享 Session State：上游 Agent 把结果写入 state['xxx']，下游 Agent 通过模板 {{xxx}} 读取。

2.6 A2A：Agent-to-Agent 协议

A2A 是 Google 主导的开放协议（已捐献给 Linux Foundation），目标让不同框架、不同组织实现的 Agent 能互相协作。核心概念：

  • Agent Card：Agent 的“名片”，描述名称、URL、能力、Skills、输入输出模式、认证方式，通常位于 /.well-known/agent-card.json。
  • Task：一次 Agent 间任务，具有生命周期（submitted → working → input-required → completed / failed）。
  • Message / Artifact：任务中的消息与产物。
  • 传输方式：JSON-RPC over HTTP；支持 Polling、Streaming（SSE/WebSocket）、Push Notification（WebHook）。

ADK 中暴露远程 Agent 使用 to_a2a(agent, port=...)；消费远程 Agent 使用 RemoteA2aAgent(agent_card=...)。这样外部专家实现的 Agent 只要提供标准 Agent Card，就能被主 Agent 发现并调用。

2.7 部署与运行时隔离

ADK 官方支持多种部署方式：

  • 本地开发：adk run / adk web / adk api_server。
  • 容器化：自定义 Dockerfile，可运行在任何支持容器的平台。
  • Google Cloud：Cloud Run、GKE、Vertex AI Agent Engine / Agent Runtime。
  • 第三方平台：LangSmith Agent Server 等。
与“每个 Agent 一个 Pod”诉求的匹配：ADK 本身不限制进程边界。我们可以把每个 Agent 单独打包镜像，通过 Kubernetes Deployment 独立运行，再用 A2A / Service Mesh 连接。平台层负责 Pod 生命周期与发现。

3.1 各层职责

主 Agent 层（ADK）

用户交互门面；理解意图；规划任务；决定调用哪些子 Agent；汇总结果；维护用户级 Session。

专业 Agent 层（ADK / 外部）

每个 Agent 独立进程/Pod；拥有独立 Skills、Tools、State；通过 A2A 对外暴露能力。

平台 Gateway 层（自研）

转发 A2A 请求；记录通信日志；统一鉴权；处理流式/异步消息；人机审批状态机。

注册与发现层（自研）

维护 Agent Card 目录；支持按 Skill 搜索；健康检查；版本管理；权重路由。

3.2 为什么必须有自研平台层？

  • 记录通信：A2A 协议本身不强制日志持久化；自研 Gateway 可以在流量入口落盘所有 Task/Message。
  • 统一发现：外部 Agent 的 Agent Card URL 可能分散在不同域名，需要注册中心聚合。
  • 安全边界：跨组织 Agent 通信需要身份校验、最小权限、TLS/mTLS、审计。
  • 人机协同：任务暂停、通知用户、接收人工输入，这些状态机不适合放在 ADK 内部。
  • 运行时编排：容器/Pod 的创建、销毁、扩缩容、网络策略属于平台层职责。

4. 详细实现计划

总周期估算：约 8～10 周（4 人小团队全职），采用“先跑通本地链路，再上平台与隔离”的渐进式策略。

Phase 0：本地 PoC（第 1～2 周）

目标验证 ADK 多 Agent + A2A 链路可行。

  • 搭建 Python 项目骨架，安装 google-adk[a2a]。
  • 实现一个“主 Agent” + 两个“子 Agent”（一个本地 sub-agent，一个通过 to_a2a() 暴露的远程 Agent）。
  • 使用 adk web 进行交互验证；跑通“用户提问 → 主 Agent 委派 → 远程 Agent 返回 → 主 Agent 汇总”的闭环。
  • 输出：可运行的 PoC 代码、问题清单、性能基线。

Phase 1：平台骨架（第 2～4 周）

目标建立 Agent 注册、发现、A2A 代理与审计能力。

  • Agent Registry：基于数据库（PostgreSQL）或 KV（Redis）实现 Agent Card 存储；提供 REST/GraphQL 查询接口。
  • A2A Gateway：FastAPI/Go 实现的透明代理，拦截 /.well-known/agent-card.json 与 /a2a/* 流量，落盘 Message/Task。
  • 主 Agent 接入注册中心：主 Agent 启动时拉取 Agent Card 列表，动态生成 RemoteA2aAgent 集合作为 sub_agents。
  • 审计查询 UI：简单的 Web 页面查看 Agent 间通信流水。

Phase 2：运行时隔离与部署（第 4～6 周）

目标让每个 Agent 可独立容器化、独立部署。

  • 为每个 Agent 编写 Dockerfile，打包成独立镜像。
  • 编写 Kubernetes manifests（Deployment + Service），每个 Agent 一个 Pod。
  • 实现基于 K8s Service / Ingress 的 Agent Card 发现，或让平台层主动注册 Pod 地址。
  • 主 Agent 运行在独立命名空间/集群，通过平台 Gateway 调用各子 Agent。
  • 引入 readiness/liveness 探针、资源限制、HPA 自动扩缩容。

Phase 3：编排增强与人机协同（第 6～8 周）

目标支持复杂 Workflow 与人工审批。

  • 规划优先模式：主 Agent 在委派前先输出执行计划（Plan-first），经用户确认或自动执行。
  • Workflow 模板：将常见模式封装为 SequentialAgent / ParallelAgent / LoopAgent 模板（会诊、报告生成、迭代优化）。
  • Human-in-the-loop：通过 LongRunningFunctionTool 或平台 Gateway 的状态机暂停任务，等待人工输入后继续。
  • 上下文压缩：启用 EventsCompactionConfig，并自定义 Summary 策略，防止长会话爆上下文。

Phase 4：安全、观测与生产化（第 8～10 周）

目标让系统可稳定运行并持续评估。

  • 安全：Gateway 层统一 OAuth2/mTLS；Agent Card 声明所需认证方式；敏感数据脱敏。
  • 观测：OpenTelemetry 链路追踪、结构化日志、Prometheus 指标、Langfuse/Phoenix 等 LLM 观测平台对接。
  • 评估：建立 Agent 评估数据集，用 ADK Eval 或自定义指标评估主 Agent 路由准确率、子 Agent 完成质量。
  • CI/CD：每个 Agent 独立仓库/流水线，容器镜像自动构建、版本化、灰度发布。

5. 风险与应对

风险影响应对策略
ADK A2A 集成仍处 Experimental API 可能变动，存在兼容性问题 封装 A2A 调用层，隔离 ADK 具体实现；关注官方 release note；必要时直接基于 a2a-sdk 实现
上下文压缩效果不稳定 长链路丢失关键信息 自定义压缩策略；保留用户原始请求、系统提示、工具链等关键消息；人工评估压缩质量
Agent 间通信延迟高 用户体验下降 本地 Agent 走进程内调用；远程 Agent 走 A2A；使用流式响应减少等待感；对慢 Agent 设置超时/异步
外部 Agent 不可控 输出质量、安全、稳定性不可控 严格 Agent Card 审核；输出校验与沙箱；超时重试；用户显式授权调用外部 Agent
平台层成为单点故障 Gateway 故障导致全系统不可用 Gateway 多副本部署；注册中心高可用；主 Agent 对本地关键能力做短路降级

6. 已确认的关键决策

经与用户确认，以下决策将指导后续实现：

  1. 通信记录范围：不记录 Agent 间完整 Message 正文；仅在主 Agent 侧记录 Task 级元数据（调用对象、状态、返回摘要），降低存储与合规成本。
  2. 隔离级别：不同 Agent 运行在不同 Container 中，通过 HTTP/A2A 通信；独立 Pod 与否保持灵活，本地调试和部署以独立 Container 为主。
  3. 外部接入标准：强制使用 A2A 协议，保证开放性与生态兼容性。
  4. 主 Agent 自主性：模型自主决策为主，预定义 Workflow 模板为辅的混合模式。
基于上述决策，平台层无需建设“全量 Agent 间消息审计”系统，重点转向 Agent 注册发现、A2A 路由转发、鉴权与观测。如后续合规要求变化，可在 Gateway 层扩展日志能力。
